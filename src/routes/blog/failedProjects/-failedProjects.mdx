## CS Serialization Experiments

If you've worked with Unity before you know that it has some weird limitations.
One big headache is serialization. Under the hood Unity uses primarily YAML to
serialize and deserialize objects, and for the most part it's pretty good at it.
Until you want to serialize something weird, like a generic class. Here's some
things I got wrong, things I got write, and things I learned.

The biggest lesson I learned from doing these kinds of experiments is not to try
to fight the tooling that you're working with, even when it's not working
exactly the way you might want. A lot of my experiments were based on saving and
loading XML strings, and a lot of them worked, but they were not very flexible
when it came to dealing with schema changes. I also made some experiments hand
rolling binary serialization solutions.

One experiment I made was for serializing objects in a raw binary file using
free lists. This experiment could dynamically allocate and free memory, but it
also required the CSharp developer to explicitically manage memory, which like...
If you're doing memory management you're loosing one of the big advantages for
working with C# in the first place.

Another experiment that I created was Hermes Serializer (no relation to the JS
compiler for React Native), a serializer that could serialize objects to a bit
stream and send them over the network. It was designed to be very inflexible,
expecting the server and client to agree ahead of time on exact definitions for
data. The idea was you would autogenerate a unique serializer/deserializer for
each type in your project for each release. Then versions of your netcode were
only compatible with exactly the same version of your netcode. But it wasn't as
reliable as I hoped.

In retrospect, it would have been better to just use fat structs, or unity
components to solve a lot of these problems, and when I'm working with these
kinds of problems today that's how I solve them. For Hermes, Protobuffs are
probably a better solution Sometimes you're just being too clever.

## Factory

Factory was my first attempt at building a programming language. Factory was, if
I'm being technical, a Domain Specific Language. It wouldn't be correct to call
it a programming language because it's not turing complete.

Factory was designed to solve logistics problems that you would run into when
playing the game Satisfactory, mainly that it could solve recipe balances for
complicated recipes.

A simple program might have looked like this. It would print a tree listing
every material needed to craft ReinforcedIronPlates.

```
print ReinforcedIronPlates(limit 60 IronOre)
```

or a more complex one might look like this. It's the same as before, but it will
use the CastScrews alternate recipe, and start with IronIngots instead of
IronOre.

```
recipe IPlates
  out ReinforcedIronPlates
  in IronIngots
  alt CastScrews
  limit 60 IronIngots

print IPlates()
```

This language had a type system, and a language server with syntax highlighting,
symbol resolution, documentation popups, and some primitive autocompletion.

If I were designing this now I would have add delimiters to my grammar. But
overall I think it was a success.

I also would probably design this as a library for another language rather then
a language itself, and maybe create a language to wrap that library.

## DnD Quest Log V3

I still pick at this project from time to time. It's much more complicated then
the V2. It looks slicker, has a nicer interface with a side bar, has a search
feature, and supports embedded components.

The biggest problem with this was that I over engineered portions of it. For one
thing, I wanted to build a universal React architecture I could use for future
projects, but that was probably a fools errand. It was miserable to work on.

I also tried to tack a mapping feature onto it, but that should have just been
it's own application.

On the positive side of things, I really got good at writing react code. Even
though the universal architecture thing wasn't the best idea, it did force me to
learn how to write better React components, services, and data structures. It's
also the first time I used tailwind, prisma, tanstack query, tanstack router,
and zod. Those have become some of my favorite libraries. They just play so well
together and solve so many problems. I think what that taught me is to find
solutions that solve specific problems well instead of solutions that try to do
everything.

## Sol

Sol is a project that I will return to after I've had a little time away from it.
It's a language designed for building developer consoles for Unity and Godot
engine. My goal was to create a library that was super easy to integrate into
your game that provided an experience comparable to working with a mature
language with a full language server.

This project wasn't a failure. The Sol runtime does work. It just got stalled out
working on the UI side of things.

## Sew

Sew is a command line utility designed for string manipulation. The way I use it
most often is to transform some string in my clipboard. You can see examples at
[sewcli.com](https://sewcli.com). I would call this a project a success. I
didn't bother with a type system for this project since it's designed to be used
interactively, and I think that was a really smart choice.

The language features syntax highlighting, and has a REPL that let's you see
the output of your code in real time, while you write it.

It features the following sub projects.

- Sew Core: Runtime for the language.
- Sew CLI: The CLI tool. It has two main features.
  - I can process strings from piped input.
  - It can run in an interactive REPL mode.
- Sew JSI: WASM module for the Sew Runtime.
- Sew Site: Source code for the sew website.
